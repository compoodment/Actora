# Actora Bugs

Actual bugs only. Not design follow-through, not deferred features.
Done = delete from here + mention in changelog.

Clarification marker:
- `❓ Needs product answer` = the bug is confirmed, but the intended target behavior or fix shape is not fully anchored by the current docs/UI contract/codebase notes. Confirm with computment before implementing.
- Unmarked bugs are sufficiently anchored for a future fix pass without extra product clarification first.

Ledger structure:
- Prefer root-cause bugs as top-level bullets.
- If multiple confirmed issues are really downstream manifestations of the same underlying bug, keep the root cause as the top-level bullet and record the surface-specific failures as indented sub-bullets under it.
- Do not create a new top-level bullet for a surface manifestation that is already explained by an existing root-cause bug unless the new finding has a distinct fix path or distinct product decision surface.
- If a sub-bug later proves to have its own distinct root cause, promote it back to a top-level bug.

## ⚪ Docs / spec drift

These are real mismatches worth fixing, but they are documentation or product-contract drift, not live simulation/runtime bugs.

- **Profile doc drift:** `docs/design.md` still says Profile is "Not started implementation" and "Read-only, no interaction," but `main.py` already implements interactive category rows and drill-down popups.
- **Profile Relationships behavior mismatch:** `docs/design.md` says pressing Enter on Relationships should open the Relationships Browser directly, but current `main.py` behavior opens a popup placeholder instead. **Superseded by card dashboard redesign (2026-04-19) — will be resolved when new Profile is implemented.**

## 🟡 Annoying

- **Lineage screen advertises a living filter hotkey that does nothing:** the lineage footer in `main.py` shows `[L] Living`, and lineage browser data supports a `living` filter, but `handle_lineage_key(...)` never handles `L`. Pressing `L` leaves `lineage_filter_mode` unchanged, so the advertised living-only filter is unreachable from the UI. ❓ Needs product answer before fix because the lineage screen itself is no longer part of the live menu flow.
- **Search-mode footers advertise `[Q] Advance`, but search handlers swallow `Q` instead:** history search, lineage search, and relationship-browser search all show footer hints that include `[Q] Advance`, but their search-mode handlers intercept input and do not route `Q` to time advancement. Pressing `Q` in those search states does nothing while the footer promises it will advance time.
- **Search UIs can claim `Esc cancels` while keeping the typed filter active:** some search handlers exit typing mode on Esc without actually clearing the stored search text, so the UI says the search was canceled while the filter remains applied. Confirmed manifestations so far include:
  - Relationships Browser search says `Type to search names. Enter confirms. Esc cancels.`, but after typing `abc` and pressing Esc, `rel_browser_search_text` still remains `abc`, so the list stays filtered
  - the orphaned lineage search says `Type to search lineage names. Enter confirms. Esc cancels search.`, but after typing `abc` and pressing Esc, `lineage_search_text` still remains `abc` and the handler only reports `Returned to lineage.`
- **Relationships Browser stored search hijacks Backspace even when search mode is closed:** the normal Browser footer says `[Bsp] Back`, but `handle_relationship_browser_key(...)` checks `rel_browser_search_text` before its normal back handling and clears the stored filter instead of leaving Browser. Reproduced with `rel_browser_search_active = False` and `rel_browser_search_text = "abc"`: pressing Backspace kept the screen on Browser, cleared the filter, and set `last_message` to `Search cleared.` instead of going back.
- **Relationships Browser actor focus needs two Backspaces to actually go back:** the Browser footer says `[Bsp] Back`, but in actor focus `handle_relationship_browser_key(...)` treats Backspace like `←` and only moves focus from actors to filters. Reproduced from Relationships Browser with actor focus active: the first Backspace kept the screen on Browser and only changed `rel_browser_focus` from `actors` to `filters`; only the second Backspace actually returned to main.
- **Skip Time cannot actually erase typed custom digits with Backspace:** the Skip Time footer promises both `[Bksp] Erase` and `[Bsp] Back`, but `handle_skip_time_key(...)` checks `key in BACK_KEYS` before its erase branch, and `BACK_KEYS` already contains the same Backspace key codes (`curses.KEY_BACKSPACE`, `127`, `8`). Reproduced by opening Skip Time, typing `123`, and pressing Backspace: the screen immediately returned to main, left `skip_custom_value` as `"123"`, and never erased a digit.
- **Hang Out target selection can queue the wrong actor when labels collide:** in `resolve_choice(...)`, the `select_hang_out_target` branch looks up the chosen target via `options_list.index(selected_value)` instead of using the popup’s current `selected_index`. If two options share the same rendered label, selecting the later one still resolves to the first matching label and queues the wrong actor.
- **Modal overlays leave stale footer hints visible, including misleading Backspace behavior:** `render_footer(...)` chooses footer text from `screen_name` but does not account for modal overlays like `pending_choice`, `menu_popup_active`, `options_popup_active`, or quit confirmation. Because the footer is still rendered under those overlays, the player can be shown underlying-screen hints like `[Bsp] Back` even when modal handling has changed Backspace to `Skip` or dialog navigation instead.
- **History year-jump scrolls to the wrong place for later years:** `jump_history_to_year(...)` finds the target year header by indexing `self.event_log`, then reuses that event index to slice `get_history_lines(...)`. But `get_history_lines(...)` inserts extra blank lines and formatted header lines, so event indexes and rendered-line indexes diverge. In practice, jumping to Year 2 or Year 3 lands above the requested year header instead of on it.

## 🟠 Significant

- **Generated humans still use a legacy trait pool:** `human.py` assigns generated actors traits from an older pool (`Calm`, `Fussy`, `Bold`, `Shy`, `Cheerful`, `Stubborn`, `Gentle`, `Alert`), and `world.py` does the same for meeting NPCs. Current mechanics and trait-gated events use the newer 12-trait pool from `mechanics.py`. This means generated parents, newborns, and meeting NPCs can carry traits that the current simulation does not recognize correctly.
- **Generated humans only receive 3 traits instead of the current 4-trait model:** `human.py` and `world.py` sample 3 traits for generated actors/NPCs, while current codebase truth says humans carry 4 traits. This leaves generated actors under-specified even aside from the legacy-trait mismatch.
- **Family newborn generation resolves sexuality at birth:** `_create_family_child_birth(...)` in `world.py` creates newborns with `randomize_stats=True`, and `Human.randomize_starting_statistics()` in `human.py` assigns a concrete sexuality immediately. Result: infants can spawn at age 0 months with sexuality already resolved, which conflicts with the current emergence model used elsewhere. ❓ Needs product answer before fix because current notes do not fully settle whether all generated NPC humans, including newborns, should auto-resolve sexuality immediately or defer it.
- **Monthly family event for children is unreachable:** `adult_checked_on_family` in `events.py` declares `family_roles=["mother", "father", "sibling", "child"]`, but `World._build_family_event_context_for(...)` in `world.py` only supplies parents and siblings. No child entries are ever provided to the event selector, so the child branch of that event can never fire. ❓ Needs product answer before fix because the mismatch is clear, but current docs do not say whether child-context support should be added or the event definition should drop child eligibility.
- **Monthly event cooldown resets every turn instead of persisting across play:** `World.simulate_advance_turn(...)` creates `recent_event_ids = deque(maxlen=3)` as a local variable on each call, so the cooldown only works within a single multi-month skip. In ordinary month-by-month play, the same monthly event can repeat on consecutive turns because prior event IDs are discarded at the start of each call.
- **Relationship Browser `Friends` filter includes plain acquaintances:** `get_relationship_entries_for(..., filter_mode="friends")` includes every active social link and only excludes `former` ones, so actors labeled `Acquaintance` still appear under the `Friends` filter. Reproduced with one active social link at closeness 15: the browser entry was labeled `Acquaintance`, but the `Friends` filter still returned it.
- **Hang Out uses an all-active social picker instead of a friend-only picker:** `open_hang_out_select()` collects every active social link without checking closeness tier, so the Social action can offer options like `Daniel Anderson · Acquaintance` even though current docs describe Hang Out as a friend picker and other UI copy says `Spend time with friend`.
- **Archived social ties still render as `Friend`/`Acquaintance` instead of `Past`:** `_build_relationship_social_entry(...)` always derives `relationship_label` from closeness via `_get_social_tier_label(...)`, even when `metadata.status == "former"`. That means the `Former`/`Past` filter and selected-person detail can show labels like `Friend` for archived ties, even though current docs say the player-facing label for former social relationships is `Past`. Reproduced with a former link at closeness 48: the `former` filter returned `Jessica Miller · Friend` with `social_status = "former"`.
- **Relationship Browser detail can contradict the selected social entry for the same actor:** `get_relationship_browser_state()` builds the visible list from the current filter, but then resolves the right-hand detail via `get_relationship_detail_for(actor_id, linked_actor_id)`, which re-queries unfiltered `all` entries and picks the first matching actor. If someone is both a family link and a social link, selecting them under `Friends` can show a left-side row like `Anthony Player · Friend`, while the detail panel shows `Father` with no closeness or social status. Reproduced by adding an active social link to the focused actor's father: the `friends` entry was `social/Friend`, but the selected detail resolved as `family/Father`.
- **Relationship Browser `All` and `Living` filters collapse multi-link actors down to family-only entries:** `get_relationship_entries_for(...)` adds family entries first, marks those actor IDs as seen, and then skips any social link for the same actor in `all` and `living` views. Reproduced by adding an active social link to the focused actor's father: `friends` returned `social/Friend` with closeness 65, but both `all` and `living` returned only `family/Father` with no social status or closeness. ❓ Needs product answer before fix because the loss is real, but current docs do not fully specify how multi-role actors should be represented in mixed filters.
- **Relationship Browser actor rows omit closeness and living/dead state promised by the screen contract:** current `screens.md` says the Relationships actor list should show name, relationship type plus closeness for social entries, living/dead state, and age. But `render_relationship_browser()` feeds every entry through `build_lineage_row(entry)`, which renders only `full_name · relationship_label · Age`. Reproduced with a social entry payload containing `closeness = 65` and `is_alive = True`: the rendered row was just `Benjamin Jackson · Friend · Age 8`. A dead family entry likewise rendered only `Joseph Player · Father · Age 30`, with no dead marker.
- **Continuation handoff exists only in shell history and leaves no preserved world record:** `choose_continuation()` appends a shell-only `life_separator` entry like `New Life: Fatima Player`, but `World.handoff_focus_to_continuation(...)` writes no corresponding `world.add_record(...)` entry. Reproduced by killing the focused actor and continuing: `event_log` gained `New Life: ...`, while `world.records` gained nothing. ❓ Needs product answer before fix because the omission is real, but current docs do not fully specify whether continuation boundaries should be preserved as inspectable simulation records, shell-history-only separators, or both.
- **A fresh life starts with no visible birth record or opening history entry:** startup player creation uses `birth_record_type="family_bootstrap"` with bootstrap wording instead of a normal `birth` record, `HIDDEN_PLAYER_RECORD_TYPES` hides that record from player-facing record surfaces, and `event_log` is not seeded from existing world records. Reproduced on a fresh game: the player's stored records were only `actor_entry` and `family_bootstrap`, `get_history_lines(...)` returned `No events yet.`, and player inspect had no recent records. ❓ Needs product answer before fix because the omission is real, but current docs do not fully settle whether the start-of-life moment should surface as a true birth record, a differently phrased startup record, or be intentionally absent from player-facing history.
- **Continuation History does not load the successor's prior life records:** `choose_continuation()` keeps or appends shell `event_log` entries, but it never rebuilds History from the new focused actor's existing `world.records`. Reproduced by giving the future father visible `birth` records before the handoff, then continuing as him: his stored records included entries like `Felix Player was born into the family.` and `Gabriela Player was born into the family.`, but after continuation `event_log` contained only `New Life: Jacob Player`, and History rendered only that separator. ❓ Needs product answer before fix because the omission is real, but current docs do not fully settle whether post-continuation History should become the successor's own prior life log, remain a cross-life saga log, or merge both views.
- **Startup player arrival is hidden from related actors' visible Recent Records:** startup player creation writes a shared `family_bootstrap` record to the player and parents, but `HIDDEN_PLAYER_RECORD_TYPES` hides that record from player-facing inspect surfaces. Reproduced by opening the father's relationship detail right after startup: the raw records were `family_bootstrap` plus `actor_entry`, but `build_record_summary_lines(...)` rendered only `No records found.` This means the parents have no visible recent-record trace that the startup child arrived, unlike ordinary later births which surface as visible `birth` records. ❓ Needs product answer before fix because the gap is real, but the right treatment depends on whether startup-family bootstrapping should become a visible birth-style record, a differently worded visible startup record, or stay hidden by design.
- **Creation stat normalization gives missing memory an incorrect positive-only default range:** `wizard.normalize_creation_stats(...)` fills missing `memory` with `random.randint(40, 70)`, even though `memory` is a signed stat whose normal baseline elsewhere is around `-10..20` or `0`. This means any startup or loaded stat block missing `memory` gets silently inflated into a strong positive-memory value, diverging from the actual stat model.
- **Profile relationship summary always undercounts friends:** `build_profile_summary_rows(...)` in `main.py` derives the Relationships summary entirely from `snapshot_data["relationships"]`, but that snapshot only contains living family entries from `Human.get_snapshot_data(...)`. Active social links are shown elsewhere in the UI, yet the Profile summary still reports `0 friends`, so the count is structurally wrong whenever the player has social connections.
- **Profile `Relationships` drill opens a dead-end popup instead of the Relationships Browser:** docs in `design.md` and `screens.md` say the Relationships row should open the Relationships tab in Browser, but `handle_profile_key(...)` always opens a generic popup for every category, and `build_profile_popup_lines("relationships", ...)` only returns `Open the Relationships Browser to see your full list.` Reproduced by opening Profile, selecting the `Relationships` row, and pressing Enter: `screen_name` stayed `profile`, `profile_popup_open` became `True`, and the popup content was just that instruction string instead of navigating to Browser.
- **Life View can list the same person twice when they are both family and social:** `build_main_left_lines(...)` renders family entries from `snapshot_data["relationships"]` and then blindly appends active social links afterward, with no dedupe or merged labeling for multi-link actors. Reproduced by adding an active social link to the focused actor's father: the left panel showed both `Ethan Player · Father` and `Ethan Player · Friend`. ❓ Needs product answer before fix because the duplication is real, but the right presentation for multi-role actors, merge into one line, prioritize one label, or show combined labels, is not fully specified.
- **Life View Relationships can claim `No living family.` while still listing active friends in the same section:** `build_snapshot_sections(...)` uses `snapshot_data["relationships"]` to populate the Relationships section and falls back to `No living family.` when no living family entries exist, while `build_main_left_lines(...)` later appends active social links underneath. Reproduced after killing all family members and adding one active friend: the section rendered `No living family.` followed by `Nicholas Miller · Friend`, which contradicts the combined family-plus-social section design.
- **Adult continuation reset leaves contradictory identity-choice flags:** after `choose_continuation()` resumes as an adult, the code sets `gender_choice_offered = True` and `sexuality_choice_offered = True`, then immediately forces `self.sexuality_choice_offered = False` unconditionally. This leaves continuation state internally inconsistent right after adult auto-resolution. ❓ Needs product answer before fix because the state contradiction is real, but the intended adult-continuation identity flow is not fully specified.
- **Social continuation candidates leak internal link labels into the UI:** `_build_continuity_candidate(...)` uses a fallback label of `f"{link_type}/{link_role}"` for non-family links, so active social continuation candidates render raw internal identifiers like `social/close_friend` instead of player-facing labels such as `Close Friend`. Reproduced by creating an active social link at closeness 72, killing the focused actor, and inspecting continuation candidates: the rendered row was `Nicholas Jackson · social/close_friend · Age 7 · New York`. The same label also propagates into continuation detail.
- **Continuation empty-state copy is narrower than the actual candidate rules:** the world seam and codebase docs say continuity candidates come from living linked actors, and the continuation flow already accepts social links, but `render_continuation()` still shows `No living family members were found.` when the list is empty. That message is now misleading because the candidate space is no longer family-only.
- **The lineage screen is implemented but unreachable from the live UI:** `main.py` still contains a full `open_lineage()` flow plus lineage rendering and key handling, but there are no live call sites to open it anymore. The menu exposes only Browser, Actions, and Profile, and `open_lineage(` appears only once in the file, at its own definition. That leaves the lineage screen orphaned even though its code, footer hints, and filters still exist. ❓ Needs product answer before fix because the bug is confirmed, but the right move could be restoring the screen, folding its value into Browser, or deleting the dead path entirely.
- **Actor Recent Records can speak in first person for someone you are no longer playing:** `World.simulate_advance_turn(...)` stores player-facing event text verbatim in world records, including lines like `"You gurgle happily at a colorful toy."`, and inspect surfaces later reuse record text without perspective rewriting via `get_relationship_detail_for(...)`, `get_lineage_detail_for(...)`, and `build_actor_inspect_detail(...)`. After a continuation handoff, inspecting the dead prior life can therefore show `"You ..."` entries under that relative’s card, which misattributes perspective and makes the recorded history read as if the inspected actor were still the current player.

## 🔴 Serious

- **Dead actors keep active social-link status after death:** `World.mark_actor_dead(...)` changes structural state and writes a death record, but it does not retire or relabel the actor’s social links. After death, relationship queries can still return the dead actor with `social_status: "active"`, so the social graph no longer matches structural reality. Manifestations already confirmed from this root bug include:
  - the Life View Relationships section can still render a dead actor like `Grace Taylor · Friend` as if they were a live social contact
  - the Hang Out picker can still offer a dead actor as a valid target
  - social-link decay can still emit post-death drift events like `You and X have drifted apart.`
- **Dead friends can still "drift apart" after they die:** because social links remain active after `World.mark_actor_dead(...)`, `World.apply_social_link_decay(...)` can continue decaying closeness against dead actors and eventually emit `"You and X have drifted apart."` for someone who is already dead. This produces impossible post-death relationship drift events.
- **Hang Out picker can offer dead actors as valid targets:** `open_hang_out_select()` in `main.py` filters social links only by `metadata.status == "active"` and does not check `target_actor.is_alive()`. Because dead actors can retain active social links, the Actions flow can present a dead person in the Hang Out picker as if they were a valid social target.
- **Queued Hang Out actions can execute against dead actors:** in `advance_time(...)`, queued `spend_time` actions are applied after `world.simulate_advance_turn(...)` without re-checking whether the target actor is still alive. If a queued social target is already dead, or dies before the action resolves, the turn can still increase closeness and log `"You spent some time with X."` for a dead actor.
- **Queued Hang Out actions can execute against actors whose social link already became `former`:** in `advance_time(...)`, queued `spend_time` actions always append the hangout event text and still award the shared happiness/stress outcome for any queued social action, even when no active social link remains by resolution time. Reproduced by queueing `Hang Out`, changing the bidirectional link to `metadata.status = "former"` before advancing, and then advancing one month: the link stayed `former`, but History still logged `You spent some time with Megan Jackson.` and the player still received `happiness +3` and `stress -2`.
- **Multiple queued Hang Out actions only award one set of mood stat effects:** in `advance_time(...)`, each queued `spend_time` action applies its own closeness gain and logs its own `You spent some time with X.` event, but the happiness/stress outcome is applied only once for the whole `spend_time_actions` batch. Reproduced by queueing Hang Out with two different active friends and advancing one month: both links gained `+5` closeness and both hangout events were logged, but the player received only `happiness +3` and `stress -2` total instead of one mood-effect application per queued action.
- **Focused actors can perform queued actions after dying in the same month:** in `advance_time(...)`, first-month queued `personal` and `spend_time` actions are resolved after `world.simulate_advance_turn(...)` returns, but before the code checks whether the focused actor died that month. This allows a dead focused actor to gain stats or log activity like `"You read something that made you think."` after death has already occurred in simulation truth.
- **Interrupted multi-month skips log the requested skip, not the months actually advanced:** `append_event_log_turn(...)` in `main.py` writes the skip marker from the requested `months_to_advance`, even when `advance_time(...)` stops early because of a popup or death. Example: a 12-month skip interrupted after 1 month still logs `"12 Months Skipped"`, which makes the event history factually wrong.
- **Continuation history inserts duplicate year headers after a life handoff:** `choose_continuation()` in `main.py` keeps the existing `event_log` but resets `last_logged_year = 0`. On the next logged event, `append_event_log_turn(...)` re-inserts `Year 1 ... Year N` headers again after the `New Life:` separator, duplicating old year headers and corrupting the chronology display.
- **Family-member deaths can happen with no player-facing notification at all:** when a non-focused family member dies through monthly mortality, the death record is written in world state, but `append_event_log_turn(...)` only surfaces birth/death records when the focused actor is in `record.actor_ids`, and `advance_time(...)` only adds special death reactions for social links. Result: close family deaths can occur silently with no event-log entry or surfaced message. ❓ Needs product answer before fix because the silence is a bug, but the exact surfacing rule, all family, close family only, caregivers only, record-only vs live message, is not fully anchored.
- **Shell-appended month events can appear before their own year header:** `advance_time(...)` directly appends queued-action, drift, and social-death-reaction log entries before it later calls `append_event_log_turn(...)`, which is the code that inserts `year_header` entries. On the first logged month of a year, this produces chronology like `"You read something..."` followed by `Year 1`, so the event log is ordered incorrectly.
- **Several player-facing life events exist only in `event_log` and never enter the world record store:** `resolve_choice(...)` logs identity decisions and meeting outcomes via `append_event_log_entry(...)`, and `advance_time(...)` does the same for queued personal actions, queued hangouts, social-death reactions, and drift events, but none of those paths write matching `world.add_record(...)` entries. As a result, the live feed and History can show things like `"You now identify as Nonbinary."` or `"You read something that made you think."` while actor inspect views, death summaries, continuation detail, and relationship/lineage Recent Records omit them entirely. ❓ Needs product answer before fix because the inconsistency is real, but current docs do not fully define which player-facing events belong in permanent world records versus live-feed-only shell history.
